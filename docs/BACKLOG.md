# Backlog: Tempus

### Épico 1: Gestão de Identidade e Perfil (Usuários)
- RF01 - Cadastro Institucional: O sistema deve permitir o cadastro apenas de usuários com e-mail @aluno.unb.br ou @unb.br.
- RF02 - Perfil de Habilidades: O usuário deve ser capaz de cadastrar e editar suas competências técnicas e teóricas (skills).
- RF03 - Gestão de Demandas: O usuário deve definir quais tipos de auxílio necessita no ecossistema acadêmico.

### Épico 2: Marketplace de Competências (Mural de Ofertas)

- RF04 - Publicação de Ofertas: O usuário deve poder publicar serviços em categorias como Acadêmico, Técnico, Consultoria, Manutenção, Assuntos e Disciplina.
- RF05 - Busca e Filtros: O sistema deve permitir a busca de ofertas por palavras-chave ou categorias
- RF06 - Mural Ativo: O sistema deve listar de forma clara as ofertas disponíveis para troca

### Épico 3: Ciclo de Vida da Troca (Sistema de Solicitação)
- RF07 - Solicitação de Serviço: Um usuário deve poder solicitar uma oferta de outro acadêmico
- RF08 - Aceite de Proposta: O prestador deve ter a opção de aceitar ou recusar uma solicitação de serviço
- RF09 - Confirmação de Entrega: Ambas as partes devem confirmar a conclusão do serviço para processar o crédito

### Épico 4: Carteira do Tempo (Carteira e Transações)
- RF10 - Gestão de Créditos: O sistema deve garantir a regra de conversão fixa: 1 hora de serviço = 1 crédito de tempo
- RF11 - Time Wallet: O usuário deve visualizar seu saldo atualizado de horas
- RF12 - Extrato de Interações: O sistema deve registrar o histórico detalhado de todas as transações, durações e datas

### Épico 5: Confiança e Governança (Reputação e Moderação)
- RF13 - Feedback Pós-Serviço: O usuário deve avaliar o serviço recebido com estrelas e comentários
- RF14 - Sistema de Disputas: Em caso de conflito na entrega, o usuário deve poder abrir um chamado de moderação
- RF15 - Painel Administrativo: A equipe Tempus deve ter interface para auditar transações e resolver disputas

### Épico 6: Colaboração Direta (Comunicação)
- RF16 - Chat de Negociação: O sistema deve prover um chat interno para que as partes alinhem escopo e horários antes do aceite
- RF17 - Notificações: O sistema deve notificar o usuário sobre novas mensagens ou atualizações em suas solicitações
